Context

문제는 단순 입력이 아니라
마감 월 기준의 원가 통제입니다

관리회계 시스템은 기준정보를 따로 유지하고, 월별로 비용을 귀속하며, 계산 결과를 다시 조회 가능한 형태로 남겨야 합니다. 이 프로젝트는 “한 달의 운영 화면”을 중심으로 등록·계산·결과 조회가 끊기지 않도록 워크스페이스 구조를 만들었습니다.

1

기준정보

본부, 부서, 직급, 인력, 프로젝트, 참여 인력을 등록합니다.

2

마감 선택

운영 콘솔은 focusedClosingPeriod 기준으로 동일한 월 데이터를 묶어 조회합니다.

4

표준원가, 공통비 배부, 내부대체가액을 순차적으로 계산합니다.

5

결과 조회

실행 버전과 결과 테이블을 운영 콘솔에서 다시 확인합니다.

Decision 01 / Web Boundary

왜 얇은 Controller + Workspace Facade 구조인가?

대안 A: Controller에서 직접 조합

화면별 요구사항을 맞추기 위해 컨트롤러가 서비스 여러 개를 직접 호출하고 조합한다.

  • 문제컨트롤러가 조합 로직을 품어 아키텍처 규칙을 쉽게 위반한다.
  • 문제마감 월 기준의 집계/정렬/변환 로직이 웹 계층에 흩어진다.
→

선택: OperationsWorkspaceFacade

컨트롤러는 요청/응답 변환만 하고, 화면 조합은 Facade가 전담한다.

  • 결과웹 계층은 redirect와 flash message 처리에 집중한다.
  • 결과focusedClosingPeriod 기준으로 기준정보·비용·결과 데이터를 한 번에 구성한다.
컨트롤러 역할등록/계산 요청을 받아 Facade 한 곳으로 위임하고, 성공/실패 notice만 구성합니다.
Facade 역할기준정보, 마감 월, 비용, 표준원가, 배부, 내부대체가액 결과를 읽어 운영 화면용 DTO로 정리합니다.

Decision 02 / Closing Control

계산은 아무 때나 돌리지 않고
마감 상태 규칙 안에서만 실행합니다

선택 기준

  • 원칙표준원가, 배부, 내부대체가액 계산 전 모두 validateCostEditable를 통과해야 합니다.
  • 상태마감 월 상태 전이는 별도 정책으로 관리하고, CONFIRMED에서는 비용 수정이 막힙니다.
  • 조회운영 콘솔은 선택 마감 월이 없으면 최신 마감 월을 기본으로 잡아 화면을 유지합니다.

Roadmap

다음 단계는 마감 결과 보존과 시연 고도화입니다

1. 기준정보·비용·계산 워크스페이스
현재 운영 콘솔에서 기준정보 등록, 비용 입력, 표준원가/배부/내부대체가액 계산이 연결되어 있습니다.
2. 웹 회귀 테스트 보강
운영 콘솔 주요 POST/GET 흐름에 대한 16개 WebMvcTest 시나리오가 추가되어 있습니다.
3. 정산 스냅샷/결과 보존
대시보드 문구 기준으로 확정 결과를 settlement 스냅샷으로 보존하는 단계가 후속 후보입니다.
4. 브라우저 기반 시연 증거 강화
보험 프로젝트처럼 실제 운영 화면 캡처와 사용자 시나리오 흐름을 포트폴리오에 추가하는 단계가 남아 있습니다.
5. 리포트/감사 관점 확장
마감 월별 결과 비교, 감사 로그, 조회 전용 화면으로 운영성을 더 강화할 수 있습니다.

Closing

이 프로젝트의 핵심 메시지

원가관리/관리회계 도메인에서 기준정보, 마감 월, 비용 귀속, 계산 결과를 하나의 운영 콘솔에 묶고, 얇은 웹 계층과 Facade 조합, 그리고 회귀 테스트로 안정성을 확보한 월마감 운영 시스템입니다.

보여준 역량

Spring Boot, Thymeleaf, JPA, 도메인 규칙 설계, 마감 상태 통제, 계산 파이프라인, WebMvcTest 기반 회귀 방어

차별점

단순 CRUD 소개가 아니라 “운영자가 월마감을 어떻게 수행하는가”를 화면 구조, 계산 흐름, 테스트 근거까지 연결해 설명한다는 점입니다.

Standalone HTML 포트폴리오 파일로 바로 열어 발표자료처럼 스크롤 시연할 수 있게 구성했습니다.
